# Git — правила для агента-разработчика

## Коммиты — только по явному указанию пользователя

- **ЗАПРЕЩЕНО** делать `git commit` без прямого указания пользователя.
- «Прямое указание» — пользователь явно сказал «коммить», «закоммить», «сделай коммит»,
  «запушь», «commit and push» и т.п.
- Не делать коммиты «на каждый чих»: после создания файла, после правки, после
  проверки — всё это НЕ повод для коммита.
- Накапливай изменения в working tree. Пользователь сам решает, когда и что коммитить.
- Если пользователь попросил «сделай X» и не упомянул коммит — выполни X и остановись.
  Не добавляй коммит «заодно».

## Push — только по явному указанию

- `git push` — только если пользователь явно сказал «пушь» / «push» / «отправь на GitHub».
- Коммит без push не означает, что нужно пушить.

## Исключения

- Если пользователь явно делегирует: «оформи всё и закоммить» — это прямое указание.
- Если в задаче прямо сказано «сделай коммит после выполнения» — это тоже прямое указание.
- Модель-участник бенчмарка следует протоколу из `AGENTS.md` — там про коммиты
  написано отдельно (модель-участник НЕ коммитит сама, см. раздел «После заполнения прогона»).

## Формат коммита

Когда коммит всё-таки делается — использовать формат из глобальных правил Devin:
- Сообщение через файл `-F` (PowerShell не поддерживает heredoc `$(cat <<'EOF'...)`)
- Тело сообщения: что + почему
- Подпись `Generated with [Devin]` + `Co-Authored-By: Devin <...>`
