# Единый чек-лист оценки прогона

Каждый пункт оценивается **0–5 баллов**:
- `0` — не выполнено / сделано хуже, чем было
- `1` — попытка есть, но результат нерабочий
- `2` — частично работает, много проблем
- `3` — в целом работает, есть замечания
- `4` — хорошо, мелкие замечания
- `5` — отлично, замечаний нет

Если пункт неприменим к категории задачи — в прогоне ставь `N/A`
(не влияет на сумму и не снижает среднее).

## Пункты

### 1. Корректность выполнения
Насколько результат соответствует постановке задачи и критериям приёмки.
Для feature/bugfix — работает ли фича/починен ли баг. Для review —
ответил ли на поставленный вопрос по существу.

### 2. Сборка и тесты
- `build_pass` = yes → +2 балла
- `tests_pass` = yes → +2 балла
- `openspec_validate_pass` = yes (если применимо) → +1 балл
Максимум 5. Если что-то из трёх неприменимо — нормируем:
например, при отсутствии openspec: build(2) + tests(2) + 1 за отсутствие регрессий = 5.

### 3. Соблюдение правил проекта
Следование `AGENTS.md` проекта: один ViewModel, токены дизайн-системы
(не inline `Color(0xFF...)`/`.dp`), миграции Room (не destructive), `safeDb`,
сортировка списков по правилам, обработка ошибок через Snackbar и т.д.

### 4. Качество кода
Идиоматичность Kotlin/Compose, корутины/StateFlow, читаемость,命名,
отсутствие мёртвого кода и избыточной вложенности.

### 5. Архитектура
Не нарушены guardrails проекта, изменения локализованы, нет leaky abstractions,
новый код в правильных пакетах/слоях. Для refactor — улучшена ли структура.

### 6. Полнота и самостоятельность
Насколько модель довела задачу до конца без необходимости дорабатывать руками.
Документация/спеки обновлены если нужно. Не оставила TODO без причины.

### 7. Объяснения и заметки
Качество собственных заметок в прогоне: что сделано, что не получилось,
какие trade-offs, что бы улучшить. Для review — глубина анализа и
обоснованность рекомендаций.

## Итог

- `total_score` = сумма баллов по применимым пунктам (N/A исключаются).
- В leaderboard используется средний балл по применимым пунктам
  (нормализованный к 5) — чтобы задачи с разным числом применимых пунктов
  были сравнимы.
