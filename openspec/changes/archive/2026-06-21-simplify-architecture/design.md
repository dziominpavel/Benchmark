## Context

Текущая архитектура `benchmark-system-core` (заархивирована) построена на концепции wave — квартального полного перезапуска всех моделей. Это требует ~184 сессий LLM на прогон (5 участников × 8 задач × 3 судьи + генерация + resolve), что нереалистично для одного человека. Дополнительно: git-теги в чужих проектах, папки `snapshots/`, `waves/`, снапшоты — всё это добавляет сложность без пропорциональной ценности.

Пользователь хочет:
- Базовый прогон за 10-15 сессий (не 184)
- Инкрементальное добавление новых моделей за 4-5 сессий
- Не править чужие проекты (никаких git-тегов)
- Задачи генерирует AI (не вручную)
- Объективную оценку (3 судьи, анонимизация)

## Goals / Non-Goals

**Goals:**
- Сократить базовый прогон до 13 сессий (9 LLM + 4 технических)
- Сократить инкрементальное добавление модели до 4-5 сессий
- Убрать зависимость от git-тегов в проектах — `baseline_commit` в `task.md`
- Батч-выполнение: 1 сессия участника = все задачи
- Батч-оценка: 1 сессия судьи = 2-3 участника × все задачи
- Кумулятивный leaderboard (модели добавляются со временем)
- Автоматическое обнаружение устаревших задач (`check_stale.py`)
- Готовые промпты для копипаста (`generate_checklist.py`)

**Non-Goals:**
- Не目标是 автоматизация запуска LLM-сессий (пользователь копипастит промпты в чат)
- Не目标是 поддержка нескольких generation-ов одновременно (один набор задач, растущий pool моделей)
- Не目标是 golden answer для feature/refactor/review (только bugfix)
- Не目标是 интеграция с CI/CD или облачными API

## Decisions

### D1: Задача = самодостаточная единица с `baseline_commit`

**Решение:** Каждая задача содержит `baseline_commit` в front matter. Участник делает `git checkout <commit>` в папке проекта. Никаких git-тегов, никаких `snapshots/`.

**Почему:** Git-теги требуют правки чужих проектов — это инвазивно. Commit hash — просто ссылка на состояние кода. Задача становится портативной: всё нужное в одном файле.

**Альтернативы:**
- Git-теги в проектах (текущая архитектура) — инвазивно, требует правки чужих репозиториев
- Branch в Benchmark-репозитории с копией кода — дублирование, расходится с реальным проектом

### D2: Батч-выполнение (1 сессия = все задачи участника)

**Решение:** Участник выполняет все задачи (4) в одной сессии. Промпт: "Выполни задачи F-001, B-001, R-001, V-001. Для каждой — checkout baseline_commit, выполни, сохрани прогон, cleanup."

**Почему:** Сокращает 40 сессий → 5. Современные LLM (200K+ контекст) нормально обрабатывают 4 задачи. В реальной работе модель решает несколько задач подряд.

**Альтернативы:**
- 1 задача = 1 сессия (текущая) — 40 сессий, нереалистично
- Батч по 2 задачи — компромисс, но 10 сессий вместо 5

**Риск:** Контекст смешивается, качество падает к концу. Mitigation: чёткая инструкция "закрой задачу перед следующей", cleanup между задачами.

### D3: Батч-оценка (1 сессия судьи = 2-3 участника)

**Решение:** Судья оценивает 2-3 участника × 4 задачи = 8-12 прогонов за сессию. При 5 участниках: 2 сессии на судью (2+3). При 1 новом участнике: 1 сессия на судью.

**Почему:** Сокращает 120 сессий → 6 (базовый) или 3 (инкремент). 8-12 прогонов ≈ 20-50K токенов — помещается в контекст.

**Альтернативы:**
- 1 прогон = 1 сессия (текущая) — 120 сессий
- 1 участник = 1 сессия — 15 сессий, но судья видит стиль участника на 4 задачах (bias)
- Батч 2-3 участника — баланс между количеством сессий и bias

**Риск:** Судья устаёт к концу, оценки менее внимательные. Mitigation: `aggregate.py` проверяет согласованность судей, спорные прогоны видны.

### D4: Golden answer только для bugfix

**Решение:** Golden answer создаётся только для bugfix-задач (где есть конкретные баги). Для feature/refactor/review — нет golden. Судья оценивает по критериям задачи + patch участника + тесты.

**Почему:** Для feature/refactor нет "правильного" ответа — golden бесполезен. Для review golden — просто "один из анализов". Для bugfix golden реально помогает: судья сравнивает найденные баги. Это убирает 6 сессий resolve (оставляет 1-2 для bugfix).

**Альтернативы:**
- Golden для всех (текущая) — 8 сессий resolve, 6 бесполезны
- Убрать golden полностью — судья bugfix проверяет patch сам (приемлемо, но +работа судье)

### D5: Кумулятивный leaderboard

**Решение:** `aggregate.py` не перегенерирует leaderboard, а обновляет. Новые модели добавляются в существующий рейтинг. Прогоны на старых `baseline_commit` валидны — модели сравниваются на одних задачах.

**Почему:** Инкрементальная модель — model-F добавляется через 2 недели, бегает на тех же задачах, что model-A..E. Сравнение честное (те же задачи, тот же commit).

**Альтернативы:**
- Перегенерация каждую волну (текущая) — нет накопления
- Отдельный leaderboard на generation — фрагментация

### D6: Глобальный pool судей (`judges.yaml`)

**Решение:** `judges.yaml` в корне репозитория — список моделей-судей. Не привязан к волне. `assign_judges.py` назначает 3 судей из pool для каждого прогона. Новые модели могут стать судьями (опционально).

**Почему:** При инкрементальном добавлении модели — не нужно пересоздавать judges.yaml. Pool растёт со временем.

**Альтернативы:**
- judges.yaml на волну (текущая) — пересоздание каждую волну
- judges.yaml на задачу — слишком гранулярно

### D7: `check_stale.py` — обнаружение устаревших задач

**Решение:** Скрипт проверяет `baseline_commit` задачи vs HEAD проекта. Если коммитов с baseline > N (порог), задача помечается `stale`. Не удаляет задачу, не делает прогоны невалидными — просто сигнал пользователю.

**Почему:** Проекты живые. Задача, написанная 3 месяца назад, может описывать код, который уже изменился. Пользователь решает: создать новую задачу или оставить.

**Альтернативы:**
- Ручная проверка — забывается
- Авто-удаление устаревших — потеря данных

### D8: `generate_checklist.py` — готовые промпты

**Решение:** Скрипт генерирует markdown-файлы с готовыми промптами для каждой фазы. Пользователь копипастит промпт в новый чат. Промпт включает: task-id, baseline_commit, model_id, пути к файлам.

**Почему:** 13 сессий × ручное составление промпта = утомительно. Готовые промпты = копипаст → запуск. Главное упрощение для пользователя.

**Альтернативы:**
- Ручное составление — медленно, ошибки
- Авто-запуск через API — не цель (пользователь использует IDE-чаты)

### D9: Структура `runs/<task-id>/<model>/` вместо `runs/<wave>/<model>/<task>/`

**Решение:** Прогоны группируются по задаче, не по волне. `runs/F-001-add-widget/claude-sonnet-4.5/run.md`. Аналогично `verdicts/F-001-add-widget/claude-sonnet-4.5/judge-X.md`.

**Почему:** Нет волн — нет уровня `wave/` в пути. Задача = верхний уровень. Изоляция по задаче: `isolate.py --task F-001 --except model-A`.

**Альтернативы:**
- `runs/<model>/<task>/` — группировка по модели (изоляция сложнее)
- `runs/<wave>/<model>/<task>/` (текущая) — нет волн

## Risks / Trade-offs

- **[Батч-выполнение: контекст смешивается] → Чёткая инструкция "закрой задачу перед следующей" + cleanup между задачами.** 4 задачи в одной сессии — риск, но современные LLM с 200K контекстом справляются.
- **[Батч-оценка: судья устаёт] → `aggregate.py` проверяет согласованность.** Спорные прогоны видны в leaderboard. Можно разбить на 2 сессии по 2-3 участника.
- **[Нет golden для feature/refactor] → Судья оценивает по критериям задачи + patch + тесты.** Это нормально — в SWE-bench судьи оценивают без golden.
- **[Устаревшие задачи] → `check_stale.py` сигнализирует, пользователь решает.** Прогоны на старом commit валидны (сравнение честное, если все на одном commit).
- **[Анонимизация при инкременте] → Новый участник добавляется в `anonymization.yaml` как "Участник F".** Судьи не знают кто это. Но если model-F была судьёй раньше — она знает задачи (OK, задачи фиксированы, все их видят).
- **[Батч-оценка: bias по стилю кода] → Анонимизация скрывает model_id, но стиль кода может выдать.** Это компромисс за сокращение сессий. В SWE-bench судьи видят несколько задач одного участника — нормальная практика.
- **[Миграция существующих прогонов] → Старых прогонов нет (только шаблоны).** Миграция тривиальна — удалить `runs/wave-01-2026Q3/`, пересоздать структуру.

## Workflow — подробное руководство

### Базовый прогон (5 участников, 3 судьи, 4 задачи) — 13 сессий

```
Phase 0: Подготовка (0 LLM-сессий)
  register_model.py × 5 участников + 3 судьи
  judges.yaml — 3 судьи
  assign_judges.py --init (анонимизация)

Phase 1: Генерация задач (1 сессия)
  Сессия с генератором:
    "Сгенерируй 4 задачи (1F + 1B + 1R + 1V) для VoiceMind.
     Для каждой — baseline_commit = текущий HEAD.
     Прочитай generators/_PROMPT.md."
  → 4 файла в tasks/<cat>/<id>/task.md

Phase 1.5: Resolve (1 сессия, только bugfix)
  Сессия с resolver:
    "Реши задачу B-001. Создай golden-answer/."
  → tasks/bugfix/B-001/golden-answer/

Phase 2: Выполнение (5 сессий, батч)
  Для каждого участника:
    isolate.py --task all --except <model-id>
    Сессия: "Выполни 4 задачи: F-001, B-001, R-001, V-001.
             Для каждой — git checkout <baseline_commit>,
             выполни, сохрани прогон, cleanup."
    isolate.py restore

Phase 3: Оценка (6 сессий, батч)
  assign_judges.py --distribute
  Для каждого судьи × 2 батча:
    isolate.py hide-verdicts --except <judge-id>
    Сессия: "Оцени прогоны Участников A, B по 4 задачам.
             Прочитай judges/_PROMPT.md."
    isolate.py restore-verdicts

Phase 4: Агрегация (0 LLM-сессий)
  aggregate.py → leaderboard.md
  Разобрать disputed
```

### Инкрементальное добавление (новая модель) — 4-5 сессий

```
register_model.py --id model-F
assign_judges.py --add-runs --participant model-F

Phase 2: 1 сессия
  model-F: "Выполни 4 задачи: F-001, B-001, R-001, V-001."

Phase 3: 3 сессии
  судья X: "Оцени прогоны Участника F по 4 задачам."
  судья Y: то же
  судья Z: то же

Phase 4: aggregate.py (обновляет leaderboard)

Опц.: model-F становится судьёй (+1 сессия калибровки)
```
