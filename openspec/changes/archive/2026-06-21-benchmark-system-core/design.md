## Context

Бенчмарк-система сравнивает LLM-модели на задачах из живых Android-проектов (GymProgress, VoiceMind, Wishlot — Kotlin/Compose/Room). Текущий каркас: `tasks/`, `snapshots/`, `runs/`, `waves/`, `tools/aggregate.py`, `models.yaml`, `AGENTS.md`. Структура `runs/` плоская (`<task-id>__<model-id>.md`), `models.yaml` заполняется руками, изоляции и анонимизации нет, судей нет, генерации задач нет.

Проекты-референсы — живые, меняются каждый квартал. Модели появляются новые каждый месяц/квартал. Результаты LLM недетерминированы — объективность достигается усреднением (3 судьи, медиана) и изоляцией (модели не видят чужие прогоны).

## Goals / Non-Goals

**Goals:**

- Воспроизводимость: все модели в одной волне бегут на одинаковом коде (git-теги), на одинаковых задачах, по одинаковому рубрикатору.
- Изоляция: участник/судья физически не видят чужие прогоны/вердикты во время работы.
- Анонимизация: судья не знает `model_id` участника, чтобы снизить предвзятость.
- Объективность через усреднение: 3 судьи на прогон, медиана, разброс > 2 → ручной разбор.
- Живучесть системы: модели и судьи меняются каждую волну, структура поддерживает ротацию.
- Калибровка задач: resolver решает до участников, golden answer — baseline для судьи (не истина).
- Минимальные зависимости: только Python stdlib, без PyYAML и внешних пакетов.

**Non-Goals:**

- Полная автоматизация: запуск моделей — ручной (пользователь открывает сессию, выбирает модель). Скрипты автоматизируют изоляцию, агрегацию, регистрацию, распределение судей — не запуск моделей.
- 100% анонимизация: стиль кода может выдать модель. Убираем только явные маркеры (имя, провайдер, саморефлексию).
- Реалтайм-сравнение: волна — квартальный процесс, не непрерывный.
- Тестирование моделей вне Android/Kotlin-стека: задачи привязаны к GymProgress/VoiceMind/Wishlot.

## Decisions

### D1. Изоляция через `isolate.py` (перемещение файлов), не через git-ветки

**Решение:** `tools/isolate.py hide --wave <wave> --except <model-id>` физически перемещает чужие прогоны из `runs/<wave>/` в `.isolation-stash/` (в `.gitignore`). После работы — `restore`.

**Альтернативы:**
- Git-ветки по моделям: надёжнее, но `git branch -a` показывает чужие ветки, и модель может `git checkout` чужую. Сложнее workflow.
- Прогоны вне репо во время работы: максимальная изоляция, но ручной шаг переноса.

**Почему:** Перемещение файлов — простое, надёжное, не требует git-операций. Модель работает в файловой системе, где чужих прогонов просто нет. `.isolation-stash/` в `.gitignore` — не попадает в коммиты.

### D2. Анонимизация через `anonymization.yaml` + анонимизированные копии прогонов

**Решение:** `assign_judges.py` создаёт `waves/<wave>/anonymization.yaml` (модель → «Участник A/B/C»). Судье даётся анонимизированная копия прогона: `model_id` заменён на «Участник X», вырезаны саморефлексии («как Claude, я...»). `anonymization.yaml` доступен только `aggregate.py`.

**Альтернативы:**
- Судья видит `model_id`: проще, но предвзятость (Claude хвалит Claude).
- Полная анонимизация кода: невозможно (стиль выдаёт).

**Почему:** Убираем явные маркеры — снижает предвзятость на ~80%. `anonymization.yaml` — единственный мост, контролируемый скриптом.

### D3. Golden answer как baseline, не истина

**Решение:** Resolver решает задачу до участников → `tasks/<cat>/<id>/golden-answer/{answer.md, diff.patch}`. Судья видит golden, но:
- находка В golden → точно реальная, зачтись
- находка НЕ В golden → судья сам проверяет (читает код), если реальная — зачтись

**Альтернативы:**
- Golden = истина: слабый resolver тянет вниз сильных участников.
- Нет golden, судья сам решает: субъективность, сложнее работа судьи.

**Почему:** Решает проблему слабого resolver. Судья и так читает код — доп. проверка «реален ли баг вне golden» не сильно усложняет.

### D4. Медиана 3 судей, разброс > 2 → ручной разбор

**Решение:** 3 судьи независимо оценивают. Итог = медиана по каждому пункту. Если разброс по любому пункту > 2 баллов → помечается `disputed`, пользователь разбирает руками (читает обоснования судей).

**Альтернативы:**
- Среднее: выброс тянет. A=4, B=4, C=1 → среднее 3, медиана 4.
- 4-й судья (tiebreaker): больше работы, не всегда нужен.

**Почему:** Медиана устойчива к выбросу. Ручной разбор — только когда все трое разошлись (редкий случай).

### D5. Двухслойный рубрикатор

**Решение:**
- `tasks/_CHECKLIST.md` — виден участнику, без баллов. «Что важно: корректность, сборка, правила проекта, качество кода, архитектура, полнота, заметки.»
- `judges/_RUBRIC.md` — скрыт от участника, с баллами 0–5 и индикаторами уровня.

**Почему:** Участник знает ожидания (как acceptance criteria тикета), но не может «геймить» конкретные индикаторы. Аналогия: студент знает критерии эссе, но не детальный рубрикатор профессора.

### D6. 8 задач на волну (2 на категорию)

**Решение:** 2 feature + 2 bugfix + 2 refactor + 2 review. При N=3 моделях: 24 прогона × 3 судьи = 72 вердикта на волну (раз в квартал — приемлемо).

**Почему:** 2 на категорию — если одна нерабочая, есть вторая. Не перегружено. Покрывает 4 типа способностей.

### D7. `register_model.py` для `models.yaml`

**Решение:** `python tools/register_model.py --id <slug> --name <name> --provider <org> --version <ver> [--released <date>]` — скрипт добавляет запись в `models.yaml` если ещё нет. Модель-участник НЕ сама себя пишет (нарушило бы изоляцию).

**Почему:** Не руками (ошибки в yaml), не самописцем (изоляция). Команда — быстро, воспроизводимо, проверяет уникальность `id`.

### D8. Clarifying questions без ответа

**Решение:** Участник пишет вопросы в прогоне (раздел «Вопросы по задаче»). Судья оценивает качество вопросов в рубрикаторе (входит в «полнота и самостоятельность»). Никто не отвечает участнику.

**Почему:** Все модели в равных условиях. Качество вопросов — сигнал понимания задачи. Сильная модель задаёт точные вопросы по реальным неоднозначностям.

### D9. Структура `runs/<wave>/<model-id>/<task-id>/{run.md, diff.patch}`

**Решение:** Вложенная структура по модели → задаче. `run.md` — front matter + заметки + clarifying questions. `diff.patch` — git diff изменений в проекте (для review-задач — отсутствует).

**Почему:** Естественная группировка по модели. Изоляция — `isolate.py` прячет всю папку `<model-id>` кроме текущей. Diff отделён от заметок — судья может применить patch к проекту и запустить тесты.

## Risks / Trade-offs

- **[Слабый resolver] → Golden = baseline, не истина (D3).** Судья может зачесть находку вне golden. Если судья тоже слаб — проблема судьи, не resolver. Выбор сильных судей.
- **[Стиль кода выдаёт модель] → Анонимизация только явных маркеров (D2).** Полная анонимизация невозможна. Снижаем предвзятость на ~80%, не на 100%.
- **[Модель нарушает запрет читать чужие папки] → Изоляция через перемещение файлов (D1).** Файлов просто нет в системе во время работы. Не «на честность».
- **[Судьи разошлись] → Медиана + ручной разбор при разбросе > 2 (D4).** Ручной разбор — редкий случай, не нагрузка.
- **[Задача нерабочая — все 5/5 или все 0/5] → Мета-анализ после 2–3 волн.** Постфактум отбраковка. Превентивно — resolver калибрует сложность.
- **[Изоляция-сташ потерян] → `.isolation-stash/` в `.gitignore`.** Если `restore` не запущен — файлы в сташе, не потеряны. Скрипт проверяет состояние при `restore`.
- **[Недетерминированность LLM] → Усреднение 3 судей + медиана.** Не убирает недетерминизм, но снижает влияние одного судьи.
- **[Ротация судей каждую волну] → `judges.yaml` на волну.** Состав судей — снимок, не глобальный. `assign_judges.py` проверяет конфликт интересов (судья ≠ участник).

## Workflow — подробное руководство для пользователя

Ниже — пошаговое описание полного цикла волны на примере 3 моделей. Это руководство для пользователя: что вызывать, в каком порядке, какие сессии открывать.

### Пример: волна wave-01-2026Q3

**Модели:**
- `claude-sonnet-4.5` (Anthropic)
- `gpt-5` (OpenAI)
- `glm-4.6` (Zhipu)

**Проекты-референсы:** GymProgress, VoiceMind, Wishlot.

### Phase 0: Подготовка снапшота (день 0, ~1 час)

```
Шаг 0.1: Git-теги в проектах
  cd C:/projects/GymProgress
  git tag benchmark-wave-01
  git push origin benchmark-wave-01

  cd C:/projects/VoiceMind
  git tag benchmark-wave-01
  git push origin benchmark-wave-01

  cd C:/projects/Wishlot
  git tag benchmark-wave-01
  git push origin benchmark-wave-01

Шаг 0.2: Заполнить snapshots/wave-01-2026Q3/
  gymprogress.yaml    → git_tag: benchmark-wave-01, commit_hash: <SHA>, summary
  voicemind.yaml      → git_tag: benchmark-wave-01, commit_hash: <SHA>, summary
  wishlot.yaml        → git_tag: benchmark-wave-01, commit_hash: <SHA>, summary

Шаг 0.3: Зарегистрировать модели
  python tools/register_model.py --id claude-sonnet-4.5 --name "Claude Sonnet 4.5" --provider Anthropic --version sonnet-4.5 --released 2025-09-29
  python tools/register_model.py --id gpt-5 --name "GPT-5" --provider OpenAI --version gpt-5 --released 2025-08-07
  python tools/register_model.py --id glm-4.6 --name "GLM-4.6" --provider Zhipu --version glm-4.6 --released 2025-09-30

Шаг 0.4: Заполнить waves/wave-01-2026Q3/
  participants.yaml:
    models: [claude-sonnet-4.5, gpt-5, glm-4.6]
  judges.yaml:
    models: [claude-sonnet-4.5, gpt-5, glm-4.6]  # круговая — каждая судит двух других

Шаг 0.5: Сгенерировать карту анонимизации
  python tools/assign_judges.py --wave wave-01-2026Q3 --init
  → создаёт waves/wave-01-2026Q3/anonymization.yaml:
    claude-sonnet-4.5 → "Участник A"
    gpt-5            → "Участник B"
    glm-4.6          → "Участник C"
```

### Phase 1: Генерация задач (день 1, ~2-3 часа)

8 задач: 2 feature + 2 bugfix + 2 refactor + 2 review. Для каждой задачи — 2 генератора пишут черновики независимо.

```
Пример: генерация F-001 (feature для VoiceMind)

Сессия 1 (генератор = Claude):
  Открыть новую сессию с Claude
  Промпт: "Ты — генератор задач для бенчмарка. Прочитай generators/_PROMPT.md.
           Тема: feature-задача для VoiceMind — добавить виджет на домашний экран.
           Сложность: средняя. Спеки: openspec/specs/widget/spec.md
           Проект: C:/projects/VoiceMind, тег benchmark-wave-01.
           Напиши черновик в tasks/_drafts/claude-sonnet-4.5/F-001-widget.md"
  → Claude пишет tasks/_drafts/claude-sonnet-4.5/F-001-widget.md

Сессия 2 (генератор = GPT-5):
  Открыть новую сессию с GPT-5
  Тот же промпт, но: "Напиши черновик в tasks/_drafts/gpt-5/F-001-widget.md"
  → GPT-5 пишет tasks/_drafts/gpt-5/F-001-widget.md

Отбор (ты):
  Читаешь оба черновика. Выбираешь лучший (или просишь одну модель
  слить лучшие части). Копируешь в tasks/feature/F-001-add-widget/task.md

Повторить для: F-002, B-001, B-002, R-001, R-002, V-001, V-002
  (16 сессий генерации: 8 задач × 2 генератора)
```

### Phase 1.5: Resolve (день 2, ~2-3 часа)

Для каждой отобранной задачи — resolver решает её до участников.

```
Пример: resolve F-001

Сессия (resolver = GPT-5, НЕ Claude — Claude был генератором F-001):
  Открыть сессию с GPT-5
  "Ты — resolver. Прочитай AGENTS.md. Реши задачу F-001
   в C:/projects/VoiceMind на теге benchmark-wave-01.
   Сохранить: tasks/feature/F-001-add-widget/golden-answer/answer.md
              tasks/feature/F-001-add-widget/golden-answer/diff.patch
   Сообщи: сложность (1-3) и время решения."
  → GPT-5 решает, сохраняет golden answer, сообщает сложность 2/3, 25 мин

Калибровка (ты):
  Сложность 2/3, 25 мин — нормальная. Задача в волну.

Повторить для всех 8 задач
  (8 сессий resolve, resolver чередуется — не та же модель что генератор)
```

### Phase 2: Выполнение (день 3-5, ~6-8 часов)

3 участника × 8 задач = 24 сессии. Каждая изолирована.

```
Пример: участник Claude выполняет F-001

Шаг 2.1: Изоляция
  python tools/isolate.py hide --wave wave-01-2026Q3 --except claude-sonnet-4.5
  → runs/wave-01-2026Q3/gpt-5/ и runs/wave-01-2026Q3/glm-4.6/ перемещены в .isolation-stash/

Шаг 2.2: Checkout тега
  cd C:/projects/VoiceMind
  git checkout benchmark-wave-01

Шаг 2.3: Сессия (участник = Claude)
  Открыть сессию с Claude
  "Выполни задачу F-001 на волне wave-01-2026Q3.
   Прочитай AGENTS.md. Файл задачи: tasks/feature/F-001-add-widget/task.md"
  → Claude читает AGENTS.md + задачу + _CHECKLIST.md
  → выполняет в C:/projects/VoiceMind
  → пишет runs/wave-01-2026Q3/claude-sonnet-4.5/F-001-add-widget/run.md
  → пишет runs/wave-01-2026Q3/claude-sonnet-4.5/F-001-add-widget/diff.patch

Шаг 2.4: Очистка проекта
  cd C:/projects/VoiceMind
  git checkout . && git clean -fd
  (снапшот чист для следующей модели)

Шаг 2.5: Восстановление изоляции
  python tools/isolate.py restore
  → чужие папки возвращены из .isolation-stash/

Повторить для: Claude × 8 задач, GPT-5 × 8 задач, GLM-4.6 × 8 задач
  (24 сессии выполнения)
```

### Phase 3: Оценка (день 6-8, ~8-10 часов)

24 прогона × 3 судьи = 72 вердикта. Судьи анонимизированы и изолированы.

```
Шаг 3.1: Распределение судей
  python tools/assign_judges.py --wave wave-01-2026Q3 --distribute
  → для каждого прогона назначает 3 судей (не та же модель что участник)
  → создаёт анонимизированные копии прогонов
  → распределение:
    Прогон Claude F-001 → судьи: GPT-5, GLM-4.6, (нужен 3-й — конфликт!
      Claude не может судить себя. При 3 моделях круговая:
      Claude судит GPT-5 и GLM-4.6, GPT-5 судит Claude и GLM-4.6,
      GLM-4.6 судит Claude и GPT-5. Каждый прогон судят 2 модели.
      → НУЖНО ≥ 4 модели для 3 судей без конфликта, ИЛИ
      → принять 2 судьи при 3 участниках, ИЛИ
      → добавить 4-ю модель только как судью)

  РЕШЕНИЕ: при 3 участниках — 2 судьи на прогон (медиана 2 значений,
  при разбросе > 2 — ручной разбор). При 4+ участниках — 3 судьи.

Шаг 3.2: Оценка (пример — GPT-5 судит прогон Claude F-001)

  python tools/isolate.py hide-verdicts --wave wave-01-2026Q3 --except gpt-5

  Сессия (судья = GPT-5):
    Открыть сессию с GPT-5
    "Ты — судья. Прочитай judges/_PROMPT.md и judges/_RUBRIC.md.
     Задача: tasks/feature/F-001-add-widget/task.md
     Прогон Участника A: <анонимизированная копия run.md>
     Diff: <diff.patch>
     Golden answer: tasks/feature/F-001-add-widget/golden-answer/
     Оцени по рубрикатору. Каждый балл — с обоснованием.
     Напиши вердикт в verdicts/wave-01-2026Q3/F-001__Участник-A/gpt-5.md"
    → GPT-5 читает всё, оценивает, пишет вердикт с обоснованием

  python tools/isolate.py restore-verdicts

Повторить для всех 24 прогонов × 2 судьи = 48 сессий оценки
  (при 3 участниках; при 4+ — 24 × 3 = 72)
```

### Phase 4: Агрегация (день 9, ~30 мин)

```
Шаг 4.1: Сгенерировать leaderboard
  python tools/aggregate.py
  → читает verdicts/wave-01-2026Q3/**/*.md
  → медиана по судьям для каждого пункта
  → деанонимизация через anonymization.yaml
  → помечает disputed (разброс > 2)
  → вычисляет качество судей (согласованность)
  → генерирует leaderboard.md

Шаг 4.2: Разобрать disputed (ты)
  Читаешь leaderboard.md, находишь ⚠ disputed
  Читаешь обоснования судей в verdicts/
  Выносишь решение, обновляешь вердикт (или оставляешь медиану)

Шаг 4.3: Заполнить summary.md (ты)
  waves/wave-01-2026Q3/summary.md:
    • Снапшоты проектов (тег + commit)
    • Состав волны: участники, судьи, генераторы, resolver'ы
    • Задачи (8) с привязкой resolver и генераторов
    • Рейтинг участников (из leaderboard)
    • Спорные прогоны (из disputed)
    • Качество судей (согласованность)
    • Выводы и рекомендации для wave-02
```

### Phase 5: Мета-анализ (после 2-3 волн, ~1 час)

```
После wave-03:
  Сессия (мета-аналитик = любая сильная модель):
    "Проанализируй результаты 3 волн.
     Данные: leaderboard.md (3 волны), verdicts/, summary.md каждой волны.
     Найди: неразличающие задачи, шумящие критерии, нестабильные модели.
     Напиши в meta-analysis/2026-12-01-summary.md"
  → рекомендации для wave-04: какие задачи заменить, критерии уточнить,
    модели пересмотреть
```
