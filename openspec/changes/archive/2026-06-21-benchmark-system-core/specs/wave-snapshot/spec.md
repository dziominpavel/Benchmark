## ADDED Requirements

### Requirement: Квартальная волна

Система MUST организовывать сравнение моделей волнами с периодичностью раз в квартал. Каждая волна имеет уникальный идентификатор формата `wave-NN-YYYYQn` (например `wave-01-2026Q3`).

#### Scenario: Идентификатор волны уникален

- **WHEN** создаётся новая волна
- **THEN** её идентификатор не совпадает ни с одной существующей волной в `snapshots/` и `waves/`

#### Scenario: Папка волны

- **WHEN** волна `wave-NN-YYYYQn` создана
- **THEN** существуют папки `snapshots/wave-NN-YYYYQn/`, `runs/wave-NN-YYYYQn/`, `verdicts/wave-NN-YYYYQn/`, `waves/wave-NN-YYYYQn/`

### Requirement: Git-тег снапшота в каждом проекте

Для каждого проекта-референса (GymProgress, VoiceMind, Wishlot) на каждой волне MUST быть поставлен git-тег `benchmark-wave-NN` в репозитории проекта. Тег фиксирует состояние кода, на котором все модели волны выполняют задачи.

#### Scenario: Тег поставлен

- **WHEN** волна `wave-01-2026Q3` подготовлена
- **THEN** в каждом проекте существует git-тег `benchmark-wave-01`
- **AND** `git checkout benchmark-wave-01` в папке проекта переводит на зафиксированный коммит

#### Scenario: Воспроизводимость

- **WHEN** две разные модели выполняют задачу на волне `wave-01-2026Q3`
- **THEN** обе работают на одном и том же коммите каждого проекта (тот же `git_tag`)

### Requirement: Снапшот-файл проекта

Для каждого проекта на каждой волне MUST существовать `snapshots/<wave>/<project>.yaml` с полями: `project`, `wave`, `captured_at`, `git_tag`, `commit_hash`, `summary`, `known_issues`, `key_docs`.

#### Scenario: Поля заполнены

- **WHEN** снапшот `snapshots/wave-01-2026Q3/voicemind.yaml` создан
- **THEN** поля `git_tag` и `commit_hash` заполнены и соответствуют тегу в репозитории проекта
- **AND** `summary` содержит краткую сводку состояния проекта

### Requirement: Состав участников волны

Волна MUST иметь файл `waves/<wave>/participants.yaml` со списком `model_id` участников. Только модели из этого списка могут выполнять задачи на волне.

#### Scenario: Участник не в списке

- **WHEN** модель с `model_id` отсутствует в `participants.yaml` волны
- **THEN** она не может создать прогон в `runs/<wave>/`

### Requirement: Состав судей волны

Волна MUST иметь файл `waves/<wave>/judges.yaml` со списком `model_id` судей. Судья не может оценивать прогоны модели, с которой совпадает по `model_id` (конфликт интересов).

#### Scenario: Конфликт интересов

- **WHEN** судья `claude-sonnet-4.5` назначен на прогон участника `claude-sonnet-4.5`
- **THEN** `assign_judges.py` отказывается назначать и выбирает другого судью из `judges.yaml`

### Requirement: Карта анонимизации волны

Волна MUST иметь файл `waves/<wave>/anonymization.yaml`, отображающий `model_id` участника в анонимный идентификатор («Участник A», «Участник B», ...). Этот файл SHALL быть доступен только скрипту `aggregate.py` для деанонимизации, и MUST NOT передаваться судьям.

#### Scenario: Судья не видит карту

- **WHEN** судья получает прогон для оценки
- **THEN** в его рабочем окружении нет `anonymization.yaml`
- **AND** в прогоне `model_id` заменён на «Участник X»

### Requirement: Сводный отчёт волны с ролями

Волна MUST иметь `waves/<wave>/summary.md` — итоговый отчёт, содержащий полную картину состава волны с указанием ролей всех моделей. Отчёт SHALL включать: снапшоты проектов (тег + commit), состав участников, состав судей, генераторов задач, resolver'ов с привязкой к конкретным задачам, рейтинг участников, спорные прогоны, качество судей (согласованность), выводы и рекомендации для следующей волны.

#### Scenario: Все роли зафиксированы

- **WHEN** волна `wave-01-2026Q3` завершена и `summary.md` заполнен
- **THEN** отчёт содержит таблицу «Состав волны» с ролями: Участники, Судьи, Генераторы, Resolver'ы
- **AND** для каждой задачи указан её resolver (`model_id`)
- **AND** для каждой задачи указаны генераторы, писавшие черновики

#### Scenario: Привязка ролей к задачам

- **WHEN** пользователь читает `summary.md` волны
- **THEN** для задачи F-001 видно: кто был генератором (черновики), кто resolver, кто участник, кто 3 судьи
- **AND** для спорных прогонов указаны оценки каждого судьи и обоснование решения

#### Scenario: Рекомендации для следующей волны

- **WHEN** `summary.md` заполнен
- **THEN** секция «Выводы» содержит: какая модель лучше под какие задачи, какие задачи отбраковать, каких судей заменить (если расходятся), что нового в проектах повлияло на результаты
