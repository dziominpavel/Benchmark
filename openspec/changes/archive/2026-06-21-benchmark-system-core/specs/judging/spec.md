## ADDED Requirements

### Requirement: 3 судьи на прогон

Каждый прогон MUST оцениваться 3 судьями независимо. Судьи не видят оценки друг друга. Состав судей определяется `waves/<wave>/judges.yaml` и распределяется `tools/assign_judges.py`.

#### Scenario: Три судьи назначены

- **WHEN** прогон `runs/wave-01-2026Q3/claude-sonnet-4.5/F-001/` готов к оценке
- **THEN** `assign_judges.py` назначает 3 судей из `judges.yaml`
- **AND** ни один из них не имеет `model_id` = `claude-sonnet-4.5`

#### Scenario: Судьи не видят чужие вердикты

- **WHEN** судья `gpt-5` оценивает прогон
- **THEN** `isolate.py hide-verdicts --except gpt-5` скрывает вердикты других судей
- **AND** в `verdicts/<wave>/` доступна только папка `gpt-5/`

### Requirement: Анонимизация прогона для судьи

Судья MUST получать анонимизированную копию прогона: `model_id` заменён на «Участник X» (из `anonymization.yaml`), вырезаны саморефлексии модели («как Claude, я...»). `anonymization.yaml` не передаётся судьям.

#### Scenario: model_id скрыт

- **WHEN** судья получает прогон участника `claude-sonnet-4.5`
- **THEN** в копии прогона `model_id` = «Участник A»
- **AND** нигде в тексте нет «claude», «anthropic», «sonnet»

#### Scenario: Саморефлексия вырезана

- **WHEN** прогон содержит фразу «как Claude, я предпочитаю...»
- **THEN** анонимизированная копия содержит «как [модель], я предпочитаю...» или фраза удалена

### Requirement: Что судья получает

Судья MUST получить: файл задачи (`task.md`), анонимизированный прогон (`run.md`), `diff.patch` (для feature/bugfix/refactor), golden answer (`golden-answer/answer.md` + `diff.patch`), `judges/_RUBRIC.md`. Судья MAY применить `diff.patch` к проекту на теге волны и запустить тесты для проверки.

#### Scenario: Полный набор артефактов

- **WHEN** судья начинает оценку прогона F-001
- **THEN** ему доступны: `tasks/feature/F-001/task.md`, анонимизированный `run.md`, `diff.patch`, `golden-answer/answer.md`, `golden-answer/diff.patch`, `judges/_RUBRIC.md`

#### Scenario: Судья проверяет тесты

- **WHEN** судья сомневается в `build_pass: yes` из прогона
- **THEN** он применяет `diff.patch` к проекту на теге волны
- **AND** запускает `./gradlew test`
- **AND** если тесты падают — ставит `build_pass: no` в вердикте

### Requirement: Вердикт с обоснованием

Судья MUST создать вердикт в `verdicts/<wave>/<run-id>/<judge-id>.md` по шаблону. Каждый балл MUST сопровождаться обоснованием (текстом, не только числом). Формат: `## <Пункт>: <балл>/5` + «Обоснование: ...».

#### Scenario: Балл с обоснованием

- **WHEN** судья ставит `correctness: 4`
- **THEN** вердикт содержит «## Корректность: 4/5» и «Обоснование: <текст>»

#### Scenario: Вердикт без обоснования невалиден

- **WHEN** вердикт содержит балл без обоснования
- **THEN** `aggregate.py` помечает вердикт как `invalid` и исключает из агрегации

### Requirement: Медиана 3 оценок

Итоговый балл по каждому пункту SHALL быть медианой 3 оценок судей. Если разброс по любому пункту > 2 баллов — прогон помечается `disputed` и пользователь разбирает руками.

#### Scenario: Медиана без спора

- **WHEN** 3 судьи поставили `correctness: 4, 4, 5`
- **THEN** итоговый `correctness` = 4 (медиана)
- **AND** разброс 1 (≤ 2) — не оспаривается

#### Scenario: Спор

- **WHEN** 3 судьи поставили `correctness: 4, 4, 1`
- **THEN** медиана = 4, но разброс 3 (> 2)
- **AND** прогон помечается `disputed` в leaderboard
- **AND** пользователь разбирает, читая обоснования судей

### Requirement: Рубрикатор судьи

`judges/_RUBRIC.md` MUST содержать детальный рубрикатор: 7 пунктов (корректность, сборка/тесты, правила проекта, качество кода, архитектура, полнота, заметки) с шкалой 0–5 и индикаторами уровня для каждого балла. Для review-задач — отдельный рубрикатор (глубина, обоснованность, полнота, конкретность, приоритизация, стиль).

#### Scenario: Индикаторы уровня

- **WHEN** судья читает `_RUBRIC.md`
- **THEN** для пункта «Корректность» указано: «0 = не выполнено, 1 = попытка, ..., 5 = отлично»
- **AND** для каждого балла есть краткий индикатор

### Requirement: Конфликт интересов судьи

Судья MUST NOT оценивать прогон модели с тем же `model_id`, что у судьи. `assign_judges.py` автоматически исключает такие назначения.

#### Scenario: Судья и участник совпадают

- **WHEN** `gpt-5` и участник `gpt-5` в одной волне
- **THEN** `assign_judges.py` не назначает `gpt-5` судьёй прогона `gpt-5`
- **AND** назначает другого судью из `judges.yaml`

### Requirement: Изоляция судей

Перед оценкой судья MUST быть изолирован от чужих вердиктов. `isolate.py hide-verdicts --wave <wave> --except <judge-id>` перемещает чужие папки `verdicts/<wave>/<other-judge-id>/` в `.isolation-stash-verdicts/`.

#### Scenario: Чужие вердикты скрыты

- **WHEN** судья `glm-4.6` начинает оценку
- **THEN** `isolate.py hide-verdicts --except glm-4.6` скрывает `verdicts/<wave>/gpt-5/` и `verdicts/<wave>/claude-sonnet-4.5/`
- **AND** в `verdicts/<wave>/` остаётся только `glm-4.6/`
