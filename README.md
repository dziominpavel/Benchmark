# Benchmark

Репозиторий для сравнения LLM-моделей на задачах из живых Android-проектов
(`GymProgress`, `VoiceMind`, `Wishlot` — Kotlin/Compose/Room).

Здесь нет кода приложений — только текстовые артефакты: задачи, прогоны моделей,
вердикты судей, leaderboard. Код живёт в отдельных репозиториях по соседству
(`C:/projects/<ProjectName>`).

## Зачем

Подобрать LLM-модель под свои задачи, опираясь на воспроизводимые прогоны
в реальном коде, а не на маркетинговые бенчмарки. Модели соревнуются на одних
и тех же задачах, на одинаковом `baseline_commit`, оцениваются 3 судьями анонимно.

## Архитектура

**Без волн.** Задача = самодостаточная единица с `baseline_commit` в front matter.
Участник делает `git checkout <commit>` — никаких git-тегов в проектах.
Новые модели добавляются инкрементально, бегут на тех же задачах.
Leaderboard кумулятивный — растёт со временем.

## Пошаговая инструкция прогона

Подробная инструкция с примерами промптов и последовательностью чатов:
**[docs/workflow-guide.md](docs/workflow-guide.md)**

### Skills — автоматизация workflow

Workflow автоматизирован через Windsurf/Devin IDE skills в `.windsurf/skills/`.
Пользователь не запускает python-скрипты напрямую — всё через skills.

| Skill | Шаг | Что делает |
|-------|-----|-----------|
| `@benchmark-00-status` | — | Показывает прогресс, подсказывает следующий шаг. Доступен ВСЕГДА. |
| `@benchmark-01-gen-task` | 1 | Диалоговая генерация задачи (по одной) |
| `@benchmark-02-resolve` | 2 | Golden answer для bugfix (промпт для resolver) |
| `@benchmark-03-run <model>` | 3 | Запуск участника (изоляция + промпт) |
| `@benchmark-04-judge <judge>` | 4 | Запуск судьи (изоляция + промпт) |
| `@benchmark-05-aggregate` | 5 | Агрегация → leaderboard |
| `@benchmark-06-stale` | — | Опционально: проверка устаревания задач |

**Порядок:** 01 → 02 → 03 → 04 → 05. `00-status` — в любой момент.
**Минимальный прогон:** 8 чатов (1 IDE + 1 resolver + 3 участника + 3 судьи).

│   ├── _TEMPLATE.md          # Шаблон задачи (с baseline_commit)
│   ├── _CHECKLIST.md         # Чек-лист для участника (без баллов)
│   ├── _drafts/              # Черновики генераторов (не коммитятся)
│   ├── feature/              # F-NNN-<slug>/task.md
│   ├── bugfix/               # B-NNN-<slug>/task.md + golden-answer/ (опц.)

├── runs/                     # Прогоны участников
│   ├── _TEMPLATE.md
│   └── <task-id>/<model-id>/
│       ├── run.md            # Заметки участника + front matter
│       └── diff.patch        # git diff изменений в проекте
├── verdicts/                 # Вердикты судей
│   ├── _TEMPLATE.md
│   └── <task-id>/<participant-anon-id>/<judge-id>.md
├── generators/
│   └── _PROMPT.md            # Промпт для модели-генератора
├── judges/
│   ├── _PROMPT.md            # Промпт для судьи
│   └── _RUBRIC.md            # Детальный рубрикатор (скрыт от участника)
├── meta-analysis/            # Анализ качества задач
├── tools/
│   ├── register_model.py     # Добавить модель в models.yaml
│   ├── isolate.py            # Скрыть/вернуть чужие прогоны и вердикты
│   ├── assign_judges.py      # Распределить судей + анонимизация
│   ├── aggregate.py          # Агрегация вердиктов → leaderboard.md
│   ├── check_stale.py        # Проверка устаревания задач
│   └── generate_checklist.py # Генерация промптов для копипаста

## Роли моделей

| Роль | Промпт | Что делает | Сессий |
|------|--------|-----------|--------|
| Генератор | `generators/_PROMPT.md` | Придумывает 4 задачи за 1 сессию | 1 |
| Resolver | `AGENTS.md` → «Resolver» | Решает bugfix до участников → golden | 1 (только bugfix) |
| Участник | `AGENTS.md` | Выполняет все задачи (батч) → прогоны + diff | 1 на участника |
| Судья | `judges/_PROMPT.md` | Оценивает прогоны анонимно (батч 2-3 участника) | 1-2 на судью |

**Правило:** модель не играет 2 роли в одном прогоне (конфликт интересов).

## Workflow — базовый прогон (5 участников, 3 судьи, 4 задачи) — 13 сессий

> Все шаги выполняются через skills в IDE-чате. Пользователь не запускает
> python-скрипты напрямую. См. [docs/workflow-guide.md](docs/workflow-guide.md)
> для детальной инструкции с примерами промптов.

### Phase 1: Генерация задач (через skill)

```
@benchmark-01-gen-task
сгенерируй задачу bugfix для GymProgress, модуль WorkoutScoreCalculator
```
→ skill предлагает формулировку → "добро" → файл создан.
Повторить для каждой задачи (по одной, не батч).

### Phase 1.5: Resolve (через skill, только bugfix)

```
@benchmark-02-resolve B-001
```
→ skill готовит промпт для resolver → вставляешь в чат с resolver-моделью →
"готово" → skill проверяет golden answer.

### Phase 2: Выполнение (через skill, по 1 чату на участника)

```
@benchmark-03-run claude-sonnet-4.5
```
→ skill изолирует чужие прогоны + golden-answer → готовит промпт →
вставляешь в чат с Claude → "готово" → skill восстанавливает изоляцию.
Модель саморегистрируется через `register_model.py --auto`.

### Phase 3: Оценка (через skill, по 1 чату на судью)

```
@benchmark-04-judge qwen-3-coder
```
→ skill распределяет судей + изолирует вердикты → готовит промпт →
вставляешь в чат с Qwen → "готово" → skill восстанавливает.
Судья саморегистрируется через `assign_judges.py --add-judge`.

### Phase 4: Агрегация (через skill)

```
@benchmark-05-aggregate
```
→ skill запускает aggregate.py, показывает leaderboard, подсвечивает disputed.

### Сводная таблица сессий

| Сценарий | LLM | Тех | Всего |
|----------|-----|-----|-------|

## Инкрементальное добавление новой модели (4-5 сессий)

```
@benchmark-03-run model-F
```
→ skill изолирует, готовит промпт → вставляешь в чат с model-F → "готово".
Модель саморегистрируется автоматически.

Затем переоценка новыми прогонами существующими судьями:
```
@benchmark-04-judge qwen-3-coder
@benchmark-04-judge llama-4-70b
@benchmark-04-judge mistral-large-2
@benchmark-05-aggregate
```

Опционально: model-F становится судьёй (+1 сессия калибровки).

## Проверка устаревания задач

```
@benchmark-06-stale
```
→ skill запускает check_stale.py, показывает таблицу устаревания.

Устаревшие задачи (> 10 коммитов с baseline) — сигнал создать новую задачу.
Прогоны на устаревших задачах валидны (сравнение честное, если все на одном commit).

## Категории задач

| Категория | Префикс | Суть | Golden answer |
|-----------|---------|------|---------------|
| Feature | F-NNN | Реализовать новую фичу | нет |
| Bugfix | B-NNN | Найти и починить баг | да |
| Refactor | R-NNN | Рефакторинг/архитектура | нет |
| Review | V-NNN | Code review / план | нет |
