# Benchmark — протокол для моделей-участниц бенчмарка

> Этот файл читает LLM-модель, которая запускается на выполнение бенчмарк-задачи
> в роли **участника**. Протоколы для других ролей (генератор, resolver, судья)
> — в `generators/_PROMPT.md`, `judges/_PROMPT.md` и этом файле (раздел «Resolver»).
>
> **Workflow автоматизирован через skills** в `.windsurf/skills/`.
> Пользователь не запускает python-скрипты напрямую — всё через skills
> `@benchmark-01-gen-task` → `@benchmark-05-aggregate`. См. `docs/workflow-guide.md`.

## Роли моделей

В бенчмарк-системе 4 роли. **Модель не может играть 2 роли в одном прогоне.**

| Роль | Промпт | Что делает |
|------|--------|-----------|
| Генератор | `generators/_PROMPT.md` | Придумывает 4 задачи за 1 сессию (не решает) |
| Resolver | раздел «Resolver» ниже | Решает bugfix-задачу до участников → golden answer |
| Участник | этот файл | Выполняет все задачи → прогоны + diff.patch |
| Судья | `judges/_PROMPT.md` | Оценивает прогоны анонимно по рубрикатору (батч) |

**Правило конфликта интересов:** если модель была генератором или resolver'ом
задачи X, она не может быть участником или судьёй задачи X.

## Что это за проект

`Benchmark` — репозиторий для сравнения LLM-моделей на задачах из живых Android-проектов
(`GymProgress`, `VoiceMind`, `Wishlot` — Kotlin/Compose/Room). Здесь нет кода приложений:
только задачи, прогоны моделей, вердикты судей и leaderboard. Код живёт
в отдельных репозиториях по соседству (`C:/projects/<ProjectName>`).

**Архитектура без волн:** задача = самодостаточная единица с `baseline_commit`
в front matter. Участник делает `git checkout <commit>` — никаких git-тегов
в проектах. Новые модели добавляются инкрементально, бегут на тех же задачах.

## Перед началом прогона — обязательные шаги

1. **Зарегистрируйся сама** — если тебя нет в `models.yaml`, добавь себя через:
   ```
   python tools/register_model.py --auto --id <slug> --name "<Имя Модели>"
   ```
   `--auto` делает `provider`/`version` опциональными. Если уже зарегистрирована —
   скрипт сообщит и завершит успешно. Никогда не придумывай `model_id` сама и
   не дописывай себя в `models.yaml` напрямую — только через скрипт.

2. **Изоляция** — пользователь запускает перед твоей сессией (через skill
   `benchmark-03-run`):
   ```
   python tools/isolate.py hide --task all --except <model-id>
   ```
   Это скрывает чужие прогоны и `golden-answer/` (для bugfix-задач).
   Ты работаешь в изоляции — чужих прогонов и эталонных ответов
   в файловой системе нет. **Не пытайся искать чужие прогоны** — их нет.

3. **Прочитай задачи** — пользователь даст тебе список задач
   (например F-001, B-001, R-001, V-001). Для каждой — открой
   `tasks/<cat>/<task-id>/task.md`. Внимательно изучи: front matter,
   описание, контекст, критерии приёмки, ограничения.

4. **Зафиксируй baseline_commit** — для каждой задачи в front matter есть
   `baseline_commit`. В папке соответствующего проекта выполни:
   ```
   cd C:/projects/<Project>
   git checkout <baseline_commit>
   ```
   **Никогда не работай на произвольной ветке** — только на commit из задачи.

5. **Прочитай чек-лист** — `tasks/_CHECKLIST.md` (критерии без баллов).
   Это то, по чему тебя будут оценивать. Детальный рубрикатор с баллами
   (`judges/_RUBRIC.md`) тебе недоступен — как в реальной работе.

6. **Прочитай правила проекта** — открой `AGENTS.md` в папке проекта
   (`C:/projects/<ProjectName>/AGENTS.md`) и соответствующие `docs/` и `openspec/specs/`.
   Соблюдай правила проекта: стиль кода, архитектурные guardrails, дизайн-систему,
   сортировку списков и т.д.

## Выполнение задач (батч)

Ты выполняешь **все задачи в одной сессии** (батч-режим). Для каждой задачи:

1. Прочитай `tasks/<cat>/<task-id>/task.md`
2. `cd C:/projects/<Project> && git checkout <baseline_commit>`
3. Выполни задачу
4. Сохрани прогон: `runs/<task-id>/<model-id>/run.md` (по шаблону `runs/_TEMPLATE.md`)
5. Сохрани diff: `git diff > runs/<task-id>/<model-id>/diff.patch`
   (для feature/bugfix/refactor; для review — только run.md)
6. Очистка: `git checkout . && git clean -fd`
7. Переходи к следующей задаче

- Работай строго в папке проекта (`C:/projects/<ProjectName>`), не в `Benchmark/`.
- Соблюдай стиль кода проекта (Kotlin/Compose/Room, один ViewModel, StateFlow, корутины).
- UI-тексты — русский; код и идентификаторы — английский.
- Не ломай существующую навигацию, схему Room (без destructive migration в release),
  дизайн-систему (токены, не inline `Color(0xFF...)` / `.dp`).
- Если задача требует OpenSpec-change — пройди через `/opsx:propose` → `/opsx:apply`
  в папке проекта, как описано в его `AGENTS.md`.
- Если есть неоднозначности — запиши clarifying questions в прогон (раздел «Вопросы по задаче»).
  Никто не ответит, но судья оценит качество вопросов.

## Проверка перед заполнением прогона

По возможности запусти в папке проекта:
- Сборка: `./gradlew assembleDebug` (или `./gradlew.bat` на Windows)
- Тесты: `./gradlew test`
- OpenSpec: `openspec validate --all` (если проект использует OpenSpec)

Зафиксируй результаты в поле `build_pass` / `tests_pass` / `openspec_validate_pass`
файла прогона. Если не удалось запустить — пометь `unknown` и объясни в заметках.

## Создание файла прогона (обязательно)

После выполнения каждой задачи создай прогон:
```
runs/<task-id>/<model-id>/run.md
runs/<task-id>/<model-id>/diff.patch  (для feature/bugfix/refactor)
```
Например: `runs/F-001-add-widget/claude-sonnet-4.5/run.md`.

Используй шаблон `runs/_TEMPLATE.md`. Заполни YAML front matter:
- `task_id`, `model_id`, `date` (YYYY-MM-DD) — обязательны
- `status`: `completed` / `partial` / `failed` / `blocked`
- `build_pass`, `tests_pass`, `openspec_validate_pass`: `yes` / `no` / `unknown`
- Баллы самооценки по каждому пункту чек-листа (0–5)
- `total_score`, `max_score`, `normalized_score`
- `diff_ref` — ссылка на diff/коммит (если есть)

Ниже front matter — секции:
- **Что сделано** — конкретно, без воды
- **Что получилось хорошо**
- **Что не получилось / проблемы**
- **Trade-offs и решения**
- **Что бы я улучшила**
- **Препятствия**
- **Вопросы по задаче** — clarifying questions (если есть)

## Diff.patch изменений

Для задач категорий feature, bugfix, refactor — сохрани `diff.patch`:
```
cd C:/projects/<Project>
git diff > C:/projects/Benchmark/runs/<task-id>/<model-id>/diff.patch
```
Для review-задач `diff.patch` не создаётся (только текстовый анализ в `run.md`).

## Очистка проекта после выполнения

После сохранения прогона и diff.patch — откат изменения в проекте:
```
cd C:/projects/<Project>
git checkout .
git clean -fd
```
Проект остаётся чистым для следующей задачи. **Не пропускай этот шаг.**

После всех задач пользователь запускает:
```
python tools/isolate.py restore
```
чтобы вернуть чужие прогоны из сташа.

## После заполнения прогона

- **Не делай `git commit` или `git push` из `Benchmark/` сама** — пользователь
  сам решает, когда коммитить и пушить (см. `.devin/rules/git.md`).
- Если в папке проекта остались изменения — отмени их (`git checkout .` / `git clean -fd`).
- Сообщи пользователю: все задачи выполнены, файлы прогонов созданы.

## Категории задач

| Категория | Папка | Префикс ID | Суть |
|-----------|-------|------------|------|
| Feature | `tasks/feature/` | `F-NNN` | Реализовать новую фичу по спеке/описанию |
| Bugfix | `tasks/bugfix/` | `B-NNN` | Найти и починить баг по репорту |
| Refactor | `tasks/refactor/` | `R-NNN` | Рефакторинг/архитектура модуля |
| Review | `tasks/review/` | `V-NNN` | Code review / план без написания кода |

## Resolver — протокол для модели-resolver'а

> Resolver решает bugfix-задачу ДО участников, создавая golden answer (baseline для судьи).
> Golden answer создаётся ТОЛЬКО для bugfix. Для feature/refactor/review — нет golden.

### Перед началом

1. Убедись, что ты НЕ была генератором этой задачи (конфликт ролей).
2. `git checkout <baseline_commit>` в папке проекта (commit из `task.md`).
3. Прочитай задачу: `tasks/bugfix/<task-id>/task.md`.

### Выполнение

- Реши задачу в папке проекта на `baseline_commit`.
- Сохраняй изменения — они пойдут в golden answer.

### Создание golden answer

```
tasks/bugfix/<task-id>/golden-answer/answer.md    ← по шаблону
tasks/bugfix/<task-id>/golden-answer/diff.patch   ← git diff
```

В `answer.md`:
- **Что сделано** — какие файлы изменены, что реализовано/найдено
- **Список находок** (для bugfix) — файл:строка, описание, критичность
- **Сложность и время** — оценка 1–3, время в минутах, комментарий
- **Замечания** (опц.) — что было непонятно в задаче

### Очистка

После сохранения golden answer — откат проект:
```
git checkout . && git clean -fd
```

### Важно

- Golden answer — **baseline**, не абсолютная истина. Судья может зачесть
  находку участника вне golden если проверит и подтвердит.
- Не сообщай участникам, что ты resolver. Они не должны знать кто resolver.
- Сообщи пользователю: сложность (1–3) и время решения — для калибровки.

## Судья — краткий протокол

> Полный промпт судьи — в `judges/_PROMPT.md`. Здесь кратко.

### Перед оценкой

1. Убедись, что ты НЕ участник этого прогона (конфликт интересов).
2. Пользователь запускает `isolate.py hide-verdicts --task all --except <judge-id>`.
3. Ты получаешь: задачу, **анонимизированный** прогон (model_id = «Участник X»),
   diff.patch, golden answer (только для bugfix), `judges/_RUBRIC.md`.
4. **Не видишь:** `model_id` участника, `anonymization.yaml`, оценки других судей.

### Оценка (батч)

- За одну сессию оцениваешь 2-3 участника × все задачи (8-12 прогонов).
- Оцени по рубрикатору, каждый балл 0–5 **с обоснованием** (текстом).
- Вердикт без обоснования — невалиден, будет исключён.
- MAY применить `diff.patch` к проекту на `baseline_commit` и запустить тесты если сомневаешься.
- Для bugfix — сравнивай с golden: находка в golden → зачтись; вне golden → проверь сам.
- Для feature/refactor/review — нет golden, оценивай по критериям задачи + patch + тесты.

### Создание вердикта

```
verdicts/<task-id>/<participant-anon-id>/<judge-id>.md
```
По шаблону `verdicts/_TEMPLATE.md`. Каждый пункт: `## <Название>: <балл>/5` + `Обоснование: ...`.

## Генератор — краткий протокол

> Полный промпт генератора — в `generators/_PROMPT.md`. Здесь кратко.

1. За одну сессию создаёшь 4 задачи (1F + 1B + 1R + 1V) для одного проекта.
2. Для каждой задачи определи `baseline_commit`: `git rev-parse HEAD` в папке проекта.
3. Создай файл `tasks/<cat>/<id>/task.md` по шаблону `tasks/_TEMPLATE.md`.
4. **Не решаешь задачу** — только формулируешь.
5. Для bugfix — НЕ перечисляй конкретные баги, только «найди N багов в модуле X».
