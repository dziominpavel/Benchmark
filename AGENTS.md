# Benchmark — протокол для модели-участника

> Этот файл читает LLM-модель, которая запускается на выполнение бенчмарк-задачи.
> Следуй ему пошагово. Это единственный источник истины по процедуре прогона.

## Что это за проект

`Benchmark` — репозиторий для сравнения LLM-моделей на задачах из живых Android-проектов
(`GymProgress`, `VoiceMind`, `Wishlot` — Kotlin/Compose/Room). Здесь нет кода приложений:
только задачи, снапшоты волн, прогоны моделей и leaderboard. Код живёт в отдельных репозиториях
в соседних папках (`C:/projects/<ProjectName>`).

## Перед началом прогона — обязательные шаги

1. **Определи свой `model_id`** — открой `models.yaml` и найди свою запись.
   Если тебя там нет — попроси пользователя добавить запись перед прогоном.
   Никогда не придумывай `model_id` сама.

2. **Узнай волну** — пользователь скажет, на какой волне ты работаешь
   (например `wave-01-2026Q3`). Соответствующая папка: `snapshots/<wave>/`.

3. **Зафиксируй снапшот проекта** — открой `snapshots/<wave>/<project>.yaml`,
   найди поле `git_tag` (например `benchmark-wave-01`). В папке соответствующего
   проекта (`C:/projects/<ProjectName>`) выполни:
   ```
   git fetch --tags
   git checkout <git_tag>
   ```
   **Никогда не работай на произвольной ветке** — только на теге снапшота волны.
   Это гарантирует, что все модели в одной волне бегут на одинаковом коде.

4. **Прочитай задачу** — пользователь даст тебе путь к файлу задачи
   (например `tasks/feature/F-001-add-rest-timer-export.md`).
   Внимательно изучи: front matter, описание, контекст, критерии приёмки, чек-лист.

5. **Прочитай правила проекта** — задача ссылается на проект. Перед изменениями
   открой `AGENTS.md` в папке проекта (`C:/projects/<ProjectName>/AGENTS.md`)
   и соответствующие `docs/` и `openspec/specs/`. Соблюдай правила проекта:
   стиль кода, архитектурные guardrails, дизайн-систему, сортировку списков и т.д.

## Выполнение задачи

- Работай строго в папке проекта (`C:/projects/<ProjectName>`), не в `Benchmark/`.
- Соблюдай стиль кода проекта (Kotlin/Compose/Room, один ViewModel, StateFlow, корутины).
- UI-тексты — русский; код и идентификаторы — английский.
- Не ломай существующую навигацию, схему Room (без destructive migration в release),
  дизайн-систему (токены, не inline `Color(0xFF...)` / `.dp`).
- Если задача требует OpenSpec-change — пройди через `/opsx:propose` → `/opsx:apply`
  в папке проекта, как описано в его `AGENTS.md`.

## Проверка перед заполнением прогона

По возможности запусти в папке проекта:
- Сборка: `./gradlew assembleDebug` (или `./gradlew.bat` на Windows)
- Тесты: `./gradlew test`
- OpenSpec: `openspec validate --all` (если проект использует OpenSpec)

Зафиксируй результаты в поле `build_pass` / `tests_pass` / `openspec_validate_pass`
файла прогона. Если не удалось запустить — пометь `unknown` и объясни в заметках.

## Создание файла прогона (обязательно)

После выполнения задачи создай файл:
```
runs/<wave>/<task-id>__<model-id>.md
```
Например: `runs/wave-01-2026Q3/F-001-add-rest-timer-export__claude-sonnet-4.5.md`.

Используй шаблон `runs/_TEMPLATE.md`. Заполни YAML front matter:
- `task_id`, `model_id`, `wave`, `date` (YYYY-MM-DD) — обязательны
- `status`: `completed` / `partial` / `failed` / `blocked`
- `build_pass`, `tests_pass`, `openspec_validate_pass`: `yes` / `no` / `unknown`
- Баллы по каждому пункту чек-листа (0–5) — из `tasks/_CHECKLIST.md`
- `total_score` — сумма баллов
- `diff_ref` — ссылка на diff/коммит (если есть)

Ниже front matter — свободные заметки: что получилось, что не получилось,
что бы ты улучшила, какие препятствия встретились.

## После заполнения прогона

- **Не делай `git push` из `Benchmark/` сама** — пользователь сам решит, когда пушить.
- Если в папке проекта остались изменения — отмени их (`git checkout .` / `git clean -fd`),
  если пользователь не сказал сохранить. Снапшот должен оставаться чистым для следующих моделей.
- Сообщи пользователю: задача выполнена, файл прогона создан по пути `<путь>`.

## Категории задач

| Категория | Папка | Префикс ID | Суть |
|-----------|-------|------------|------|
| Feature | `tasks/feature/` | `F-NNN` | Реализовать новую фичу по спеке/описанию |
| Bugfix | `tasks/bugfix/` | `B-NNN` | Найти и починить баг по репорту |
| Refactor | `tasks/refactor/` | `R-NNN` | Рефакторинг/архитектура модуля |
| Review | `tasks/review/` | `V-NNN` | Code review / план без написания кода |

## Чек-лист оценки

Все задачи оцениваются по единому чек-листу — `tasks/_CHECKLIST.md`.
Каждый пункт 0–5 баллов. Итог = сумма. Максимум зависит от числа пунктов.
