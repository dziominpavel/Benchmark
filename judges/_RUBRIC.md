# Рубрикатор судьи

Каждый пункт оценивается **0–5 баллов**. Каждый балл MUST сопровождаться обоснованием.

Шкала общая:
- `0` — не выполнено / сделано хуже, чем было
- `1` — попытка есть, но результат нерабочий
- `2` — частично работает, много проблем
- `3` — в целом работает, есть замечания
- `4` — хорошо, мелкие замечания
- `5` — отлично, замечаний нет

Если пункт неприменим — ставь `N/A` с обоснованием.

---

## Рубрикатор для feature / bugfix / refactor

### 1. Корректность (0–5)

Насколько результат соответствует постановке задачи и критериям приёмки.
Для bugfix — найдены ли реальные баги (сравни с golden answer), есть ли ложные срабатывания.
Для feature — работает ли фича, соответствует ли спеке.
Для refactor — улучшена ли структура, сохранено ли поведение.

| Балл | Индикатор |
|------|-----------|
| 0 | Задача не выполнена, результат сломан |
| 1 | Попытка есть, но ничего не работает |
| 2 | Частично работает, основные функции не реализованы |
| 3 | В целом работает, пропущены детали или edge cases |
| 4 | Хорошо, мелкие замечания по edge cases |
| 5 | Все критерии приёмки выполнены, edge cases покрыты |

**Сравнение с golden:**
- Находка В golden → зачтись.
- Находка НЕ В golden → проверь сам. Реальная → зачтись, отметь «вне golden».
- Ложное срабатывание → не зачтись, снизь балл.

### 2. Сборка и тесты (0–5)

- `build_pass` = yes → +2 балла
- `tests_pass` = yes → +2 балла
- `openspec_validate_pass` = yes (если применимо) → +1 балл

Максимум 5. Если что-то неприменимо — нормируй:
- Нет openspec в проекте: build(2) + tests(2) + 1 за отсутствие регрессий = 5
- Только build (тестов нет в проекте): build(2) + 3 за качество ручной проверки = 5

**Судья MAY применить patch и запустить тесты сам** если сомневается в самоотчёте участника.

### 3. Соблюдение правил проекта (0–5)

Следование `AGENTS.md` проекта:
- Один ViewModel (не создан второй)
- Токены дизайн-системы (не inline `Color(0xFF...)` / `.dp` / `RoundedCornerShape(...)`)
- Миграции Room (не destructive в release)
- `safeDb` / обработка ошибок через Snackbar
- Сортировка списков по правилам проекта
- Архитектурные guardrails не нарушены

| Балл | Индикатор |
|------|-----------|
| 0 | Правила грубо нарушены (destructive migration, inline цвета, второй ViewModel) |
| 1 | Множественные нарушения |
| 2 | Несколько нарушений, но не критичных |
| 3 | 1–2 мелких нарушения |
| 4 | Правила соблюдены, 1 мелкая оговорка |
| 5 | Все правила соблюдены |

### 4. Качество кода (0–5)

Идиоматичность Kotlin/Compose:
- Корутины, StateFlow (не Callback Hell)
- Compose-идиомы (не императивный UI)
- Именование, читаемость
- Отсутствие мёртвого кода и избыточной вложенности
- Компактность (no boilerplate)

| Балл | Индикатор |
|------|-----------|
| 0 | Код не идиоматичный, нечитаемый |
| 1 | Множественные проблемы качества |
| 2 | Несколько проблем, но в целом читаемо |
| 3 | Идиоматично, есть мелкие замечания по стилю |
| 4 | Хорошо, 1–2 мелкие оговорки |
| 5 | Идиоматичный, чистый, компактный |

### 5. Архитектура (0–5)

- Изменения локализованы (не размазаны по кодовой базе)
- Нет leaky abstractions
- Новый код в правильных пакетах/слоях
- Для refactor — структура улучшена, не просто перемещено

| Балл | Индикатор |
|------|-----------|
| 0 | Архитектура нарушена, изменения хаотичны |
| 1 | Множественные архитектурные проблемы |
| 2 | Несколько проблем, но в целом локализовано |
| 3 | Локализовано, 1–2 замечания по слоям |
| 4 | Хорошо, мелкие замечания |
| 5 | Чистая архитектура, правильные слои |

### 6. Полнота и самостоятельность (0–5)

- Задача доведена до конца (не брошена на полпути)
- Документация/спеки обновлены если нужно
- Нет TODO без причины
- Clarifying questions — точные и по делу (если были)
- Разумные предположения при неоднозначностях

| Балл | Индикатор |
|------|-----------|
| 0 | Задача брошена, ничего не доведено |
| 1 | Большая часть не завершена |
| 2 | Частично завершена, много TODO |
| 3 | В целом завершена, мелкие пробелы |
| 4 | Завершена, 1–2 мелкие оговорки |
| 5 | Полностью завершена, осмысленные clarifying questions |

### 7. Заметки и объяснения (0–5)

Качество заметок в прогоне:
- Что сделано — конкретно, без воды
- Trade-offs — названы и обоснованы
- Что не получилось — честно, с причинами
- Что бы улучшила — конкретные предложения
- Препятствия — описаны

| Балл | Индикатор |
|------|-----------|
| 0 | Заметок нет или они бессмысленны |
| 1 | Заметки поверхностные, воды |
| 2 | Частично полезные, много пропущено |
| 3 | Полезные, но не хватает глубины |
| 4 | Хорошие, мелкие пробелы |
| 5 | Полные, конкретные, с trade-offs |

## Рубрикатор для review (V-NNN)

Review-задачи не имеют diff.patch. Участник пишет только текстовый анализ.
Сравнивай с golden answer (анализ resolver'а).

### 1. Глубина (0–5)

Нашла ли реальные проблемы в коде (совпадение с golden + находки вне golden).

| Балл | Индикатор |
|------|-----------|
| 0 | Не нашла проблем |
| 1 | Поверхностный анализ, пропущено критичное |
| 2 | Найдена часть проблем |
| 3 | Найдено большинство проблем |
| 4 | Найдены все основные + некоторые тонкие |
| 5 | Все проблемы из golden + дополнительные реальные |

### 2. Обоснованность (0–5)

Рекомендации обоснованы, не голословны.

| Балл | Индикатор |
|------|-----------|
| 0 | Рекомендации без обоснования |
| 1 | Голословные, «переписать всё» |
| 2 | Частично обоснованы |
| 3 | В основном обоснованы, есть пробелы |
| 4 | Хорошо обоснованы, мелкие пробелы |
| 5 | Каждая рекомендация с конкретным обоснованием |

### 3. Полнота (0–5)

Покрыла ли весь код/модуль, или только часть.

| Балл | Индикатор |
|------|-----------|
| 0 | Просмотрена малая часть |
| 1 | Только верхний уровень |
| 2 | Часть модуля |
| 3 | Большая часть модуля |
| 4 | Весь модуль, мелкие пропуски |
| 5 | Полное покрытие |

### 4. Конкретность (0–5)

Точечные улучшения, а не «переписать всё».

| Балл | Индикатор |
|------|-----------|
| 0 | Абстрактные рекомендации |
| 1 | «Переписать», без конкретики |
| 2 | Частично конкретные |
| 3 | В основном конкретные, с указанием файлов/строк |
| 4 | Конкретные, с примерами кода |
| 5 | Точечные, с примерами и альтернативами |

### 5. Приоритизация (0–5)

Что критично, что нет. Порядок исправления.

| Балл | Индикатор |
|------|-----------|
| 0 | Нет приоритизации |
| 1 | Всё «критично» или всё «мелочь» |
| 2 | Грубая приоритизация |
| 3 | Приоритизация есть, но не всегда точная |
| 4 | Хорошая приоритизация |
| 5 | Чёткая: critical → high → medium → low, с обоснованием |

### 6. Стиль (0–5)

Понятный, структурированный отчёт.

| Балл | Индикатор |
|------|-----------|
| 0 | Неструктурированный, непонятный |
| 1 | Хаотичный текст |
| 2 | Частично структурированный |
| 3 | Структура есть, но непоследовательно |
| 4 | Хорошо структурированный |
| 5 | Чёткая структура, заголовки, таблицы, читаемо |

## Итог

- `total_score` = сумма баллов по применимым пунктам (N/A исключаются)
- `max_score` = 5 × число применимых пунктов
- `normalized_score` = total / max × 5 (для leaderboard)
- В leaderboard — медиана по 3 судьям (или 2 при 3 участниках)
- Разброс > 2 по любому пункту → `disputed` → ручной разбор
